# EkiLog — Agent Instructions

EkiLog is a **SvelteKit PWA** that helps users log Japanese railway journeys to [Träwelling](https://traewelling.de). It scans physical tickets via Gemini AI, searches stations from a local IndexedDB, looks up HAFAS departures, and submits check-ins.

## Build & Dev Commands

```bash
npm run dev          # start dev server
npm run build        # static build → build/
npm run preview      # preview production build
npm run check        # svelte-check + TypeScript
npm run lint         # prettier + eslint
npm run format       # auto-format
```

## Architecture

- **Single-page SPA** — `ssr = false`, `prerender = true`, static adapter (GitHub Pages)
- `src/routes/+page.svelte` — tab UI: Scanner → CheckInForm → Map
- `src/lib/components/` — CheckInForm, Scanner, StationSearch, Map, Settings
- `src/lib/stores/` — Svelte 5 runes-based stores (see below)
- `src/lib/api/traewelling.ts` — Träwelling REST API client
- `src/lib/api/gemini.ts` — Gemini Flash Lite ticket parsing
- `src/lib/db/stations.ts` — IndexedDB for Japanese station data (open-data-jp)
- `src/lib/paraglide/` — **generated** i18n code, never edit directly

## Key Conventions

### Svelte 5 Runes — Always Active

Runes mode is **forced project-wide** in `vite.config.ts` (even for `.svelte` files, not just `.svelte.ts`). Never use classic Svelte reactive syntax (`$:`, `writable()`, `$store`). Always use `$state`, `$derived`, `$effect`.

### State Stores (runes pattern)

```ts
// Usage — NOT classic store syntax
import { journey } from '$lib/stores/journey.svelte';
journey.data          // read
journey.update({...}) // partial merge
journey.reset()       // clear
```

`config.svelte.ts` auto-persists to `localStorage` on every setter call.

### Internationalization (Paraglide)

- Messages live in `messages/{locale}.json`, compiled to `src/lib/paraglide/`
- Import: `import { m } from '$lib/paraglide/messages.js'` → call as `m.my_key()`
- **Never add raw string literals** for UI text — always add a message key
- Add keys to **all 5** locale files: `en`, `de`, `es`, `fr`, `ja`
- Locale changes trigger a full page reload

### daisyUI + Tailwind CSS v4

- No `tailwind.config.js` — configured via Vite plugin only.
- Adhere to daisyUI style conventions for consistent UI. Use the skill for further guidance on daisyUI components and classes.
- Use Tailwind utility classes for custom styling, but prefer daisyUI components when possible.

### API Pattern

Both API modules export standalone async functions that throw typed errors (`TraewellingError`, `GeminiError`). Loading states follow `'idle' | 'loading' | 'success' | 'error'`.

### Map Component

Leaflet is **dynamically imported** to avoid SSR errors. Always keep Map.svelte as a lazy-loaded component.

## Common Pitfalls

- **Don't edit `src/lib/paraglide/`** — regenerated by build
- **Station search is O(n)** — full in-memory scan of ~7k entries; no spatial index
- **Manual trips require `category`** — Shinkansen/non-HAFAS trains use `createManualTrip()`; the `category` enum must be set by user
- **Gemini API key is client-side** — user provides their own key; no backend
- **HAFAS departure lookup is time-window limited** — only works within ~2 hours of scheduled departure
- **`404.html` is the SPA fallback** — required for GitHub Pages routing

## Available Svelte MCP Tools:

### 1. list-sections

Use this FIRST to discover all available documentation sections. Returns a structured list with titles, use_cases, and paths.
When asked about Svelte or SvelteKit topics, ALWAYS use this tool at the start of the chat to find relevant sections.

### 2. get-documentation

Retrieves full documentation content for specific sections. Accepts single or multiple sections.
After calling the list-sections tool, you MUST analyze the returned documentation sections (especially the use_cases field) and then use the get-documentation tool to fetch ALL documentation sections that are relevant for the user's task.

### 3. svelte-autofixer

Analyzes Svelte code and returns issues and suggestions.
You MUST use this tool whenever writing Svelte code before sending it to the user. Keep calling it until no issues or suggestions are returned.

### 4. playground-link

Generates a Svelte Playground link with the provided code.
After completing the code, ask the user if they want a playground link. Only call this tool after user confirmation and NEVER if code was written to files in their project.
